  - **실수**: `clearTokenCache()`가 인메모리 토큰만 초기화하고 `kis_tokens` DB 캐시는 남겨 두어, 다음 `getAccessToken()` 호출이 오래된 DB 토큰을 다시 읽어 AUTH 오류를 반복시킬 수 있었다.
  - **원인**: 서버리스/다중 인스턴스 환경에서 공유 DB 캐시가 실제 우선 복구 경로라는 점을 토큰 무효화 로직에 반영하지 못했다.
  - **해결 및 재발 방지**: 토큰 초기화 요청은 반드시 인메모리 캐시와 `kis_tokens` DB 레코드를 함께 삭제한다. AUTH 오류 후 재시도나 수동 sync처럼 즉시 새 토큰이 필요한 경로에서는 `await clearTokenCache()`로 삭제 완료를 보장한 뒤 재발급한다.
- **클라이언트 자동 동기화(Polling)와 서버 토큰 초기화 로직 충돌로 인한 API 토큰 과다 발급 문제 방지**:
  - **실수**: 클라이언트가 10초 주기로 최신 데이터를 가져오기 위해 호출하는 `/api/stock/.../sync` 라우터 내부에, 의도와 달리 `clearTokenCache()` 토큰 삭제 로직을 무조건적으로 배치함.
  - **원인**: "데이터를 새로고침(Sync)한다"는 개념과 "인증 토큰을 초기화(Clear)한다"는 개념을 엄격하게 분리하지 못하고, Sync 라우터에 두 로직을 섞어버리는 아키텍처 설계 실수를 범함. 이로 인해 정상적으로 20시간 유지되어야 할 KIS 토큰이 10초마다 삭제 및 신규 발급되어 API Rate Limit 이슈를 유발함.
  - **해결 및 재발 방지**: 데이터 동기화(Sync) API 라우터에서는 인증 토큰 강제 삭제 로직을 완전히 제거함. 토큰 무효화 및 재발급 로직은 오직 KIS API에서 명시적으로 `AUTH` 에러나 HTTP 401 인증 실패를 반환했을 때만 스스로 작동하도록(Self-Healing 로직 내부에 격리하여) 책임을 완벽히 분리함. 향후 인증(Auth)이 포함된 외부 API 연동 시에는, **데이터 갱신 주기**와 **토큰 수명 주기(Lifecycle)**를 절대 동기화하지 않고 엄격히 분리하여 설계하는 원칙을 수립함.
